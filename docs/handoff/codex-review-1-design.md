**Findings**

- `[critical]` 메인 스펙의 클린슬레이트 범위가 FK 현실과 충돌합니다. 메인 문서는 “사용자 데이터는 루틴만 wipe”라고 적고 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:39), 실제 순서도 `workout_routines(+cascade)`만 지운 뒤 `workout_logs`/`user_exercise_1rm`은 “0행이면” 넘기려 합니다 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:137). 그런데 `routine_exercises.exercise_id/equipment_id`는 `RESTRICT` [routine.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/models/routine.py:90), `workout_log_sets.exercise_id`도 `RESTRICT` [workout.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/models/workout.py:53), `user_exercise_1rm.exercise_id`는 `CASCADE` [user.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/models/user.py:109)입니다. 즉 `exercises/equipments` wipe는 루틴만 지워서는 안 되고, 로그가 남아 있으면 삭제가 막히고, 1RM은 남기려 해도 자동 삭제됩니다. 같은 날 작성된 handoff는 이미 이 사실을 인정하고 logs/1rm까지 wipe 동의를 요구합니다 [reconciliation](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/2026-06-06-workoutx-canonical-reconciliation.md:107). 지금 스펙은 삭제 순서가 아니라 삭제 정책 자체가 모순입니다.

- `[high]` `routine_exercises.equipment_id NULL 허용 + 프리웨이트 equipment 행 제거`는 루틴 생성만 고치면 끝나는 변경이 아닙니다. 현재 저장 경로는 `equipment_id`가 없으면 운동을 버립니다 [routines.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/api/v1/routines.py:1315), 종목 교체도 자동으로 equipment를 다시 선택합니다 [routines.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/api/v1/routines.py:457). 세션/PO 경로는 `RoutineExercise.equipment_id`에서 `equipment_type/max_stack`를 읽어 중량 증가를 계산합니다 [sessions.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/api/v1/sessions.py:500), 추천 중량도 여전히 `equipment_type/pulley_ratio/bar_weight` 입력을 전제로 합니다 [routine_targets.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/services/routine_targets.py:116). 스펙은 `routines.py`, `rag.py`, `load_calc.py`만 언급하지만 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:99), 실제로는 sessions/users/PO까지 재설계 대상입니다. blast radius를 심각하게 축소 평가했습니다.

- `[high]` `load_mode` 어휘가 WorkoutX 실제 equipment 분포를 못 담습니다. 스펙 enum에는 `weighted`가 없고 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:47), 매핑 규칙은 `Weighted`를 `bodyweight`로 흡수합니다 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:124). 그런데 실제 캐시 분포에는 `Weighted` 36개, `Resistance Band` 7개, `Trap Bar` 1개, `Assisted (towel)` 1개, 각종 cardio machine이 존재합니다 [distributions.csv](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/workoutx-raw/distributions.csv:39). 예시로 `Weighted Decline Sit-up`은 실제로 WorkoutX가 `equipment: "Weighted"`로 줍니다 [exercises.json](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/workoutx-raw/exercises.json:35073). 이걸 `bodyweight` 한 축으로 뭉개면 “체중 기반+외부부하”와 “순수 맨몸”의 부하 의미가 섞입니다. `load_mode`가 새 진실원천이라면, 최소한 `weighted`를 독립시키거나 외부부하 여부를 별도 속성으로 분리해야 합니다.

- `[high]` `movement_label`을 제거하면서 대체할 “기구 선택 의미론”이 비어 있습니다. 현재 머신 경로는 `movement_label_en`이 실물 기구와 canonical exercise를 잇는 유일한 브리지입니다 [routines.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/api/v1/routines.py:1171), 후보 목록도 그 라벨을 LLM 계약으로 씁니다 [routines.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/api/v1/routines.py:935), [rag.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/services/rag.py:401). 새 `exercise_equipment`는 `source/confidence`만 있고 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:63), “이 헬스장에서는 이 운동에 어떤 기계를 우선 보여주나”, “표시 라벨은 무엇인가”, “동일 운동을 여러 장비가 지원할 때 기본 선택은 무엇인가”가 없습니다. N:M만 만들고 selection/display rank를 정의하지 않으면, persistence는 결국 임의 1개 고르기로 퇴화합니다.

- `[high]` WorkoutX taxonomy 채택의 손실을 너무 가볍게 처리합니다. `activation_pct` 병합 규칙을 `MAX`로 정하면 전/측/후삼각근 중 하나만 강한 운동이 `delts` 전체가 강한 것처럼 과대 대표됩니다 [reconciliation](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/2026-06-06-workoutx-canonical-reconciliation.md:67). 거기에 API는 값이 하나라도 `NULL`이면 과학 데이터 대신 70/30 휴리스틱으로 덮습니다 [routines.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/api/v1/routines.py:255). 더 나쁜 건 캐노니컬 20종에는 `Hip Flexors`를 넣어놓고 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:74), activation salvage에서는 `hip_flexors`를 drop 처리했다는 점입니다 [reconciliation](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/2026-06-06-workoutx-canonical-reconciliation.md:80). 실제 secondary 빈도도 77건입니다 [muscle_normalization.md](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/workoutx-raw/muscle_normalization.md:30). 이건 “손실을 감수” 수준이 아니라, 일부 핵심 보조근 데이터를 구조적으로 비워두겠다는 뜻입니다.

- `[high]` 스펙 문서들끼리 SOT가 아닙니다. 메인 설계는 `load_mode` 추가, `default_equipment_id/is_freeweight/movement_label` 제거, `routine_exercises.equipment_id` NULL 허용을 명시합니다 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:34). 그런데 reconciliation은 “DB 스키마 변경 불필요”, `equipment_type` 유지, `default_equipment_id(equipment)`로 시드하자고 씁니다 [reconciliation](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/2026-06-06-workoutx-canonical-reconciliation.md:11), [reconciliation](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/2026-06-06-workoutx-canonical-reconciliation.md:35), [reconciliation](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/2026-06-06-workoutx-canonical-reconciliation.md:121). 구현 전 설계 문서가 서로 반대 방향이면, 지금 필요한 건 마이그레이션이 아니라 SOT 정리입니다.

- `[medium]` `secondaryMuscles` 정규화 맵에 근거 빈약한 강제 매핑이 섞여 있습니다. 대표적으로 `Back -> Spine` [muscle_normalization.md](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/workoutx-raw/muscle_normalization.md:40), `Sternocleidomastoid -> Levator Scapulae` [muscle_normalization.md](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/workoutx-raw/muscle_normalization.md:48)는 해부학적으로 동치가 아닙니다. 드롭보다 낫다는 식의 임시 흡수는 가능하지만, 그런 데이터는 적어도 `normalized_from_raw` 같은 provenance 없이 canonical truth로 굳히면 안 됩니다.

- `[medium]` `equipments.category`와 `muscle_groups.name_ko`에 대한 설계가 허술합니다. 스펙 스스로 `equipments.category`를 WorkoutX bodyPart로 통일할지 “경미”하다고 쓰지만 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:164), 케이블 스테이션·트레드밀 같은 실물 기구는 단일 bodyPart 자산이 아닙니다. 동시에 `MuscleGroup.name_ko`는 unique 제약인데 [exercise.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/models/exercise.py:60), 한글명은 Gemini 일괄 번역 후 검수로 넘기겠다고만 합니다 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:38). 과거에도 `name_ko` 중복 충돌 흔적이 문서화돼 있습니다 [table-population-spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/2026-06-06-table-population-spec.md:30). 충돌 해결 규칙 없이 AI 번역을 seed truth로 쓰는 건 데이터 정합성 설계가 아닙니다.

- `[medium]` “WorkoutX 원본 캐시가 repo에 없다”는 전제가 사실과 다릅니다. reconciliation은 라이브 재수집을 전제로 캐시 부재를 주장하지만 [reconciliation](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/handoff/2026-06-06-workoutx-canonical-reconciliation.md:89), 실제로는 `docs/handoff/workoutx-raw/exercises.json`과 분포 파일이 이미 있으며 메인 스펙도 그 파일을 참조합니다 [redesign spec](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/docs/spec/2026-06-06-exercise-equipment-workoutx-redesign.md:170). 재현 가능한 frozen snapshot이 있는데도 라이브 API 재수집을 기본 전제로 두면, 구현 시점마다 운동 수/어휘가 흔들릴 수 있습니다.

논문 데이터 보존 자체는 맞는 방향입니다. `paper_chunks/papers`는 과거 clean-slate 마이그에서 실제 `DROP TABLE ... CASCADE` 대상이었고 [007 migration](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/alembic/versions/007_clean_slate_papers_multi_source.py:37), RAG는 기본 컬렉션도 `paper_chunks`를 전제로 합니다 [rag.py](/mnt/c/Users/User/Desktop/coding/Main_Project/capstone/scifit-sync/server/app/services/rag.py:57). 그래서 여기만큼은 “절대 불가침”이 과장이 아닙니다.

우선순위로 정리하면, 지금 당장 잠가야 할 건 1) wipe 범위와 FK 처리, 2) `default_equipment_id` 유지 vs `load_mode` 전환 중 하나의 SOT 선택, 3) `load_mode` 어휘 재정의(`weighted` 포함), 4) `exercise_equipment`에 selection/display semantics 추가입니다. 이 네 개가 정리되기 전에는 구현에 들어가면 거의 확실하게 재마이그가 납니다.
